# Ember — Architecture & Theme Audit

_Date: 2026-06-29 · Scope: `artifacts/ember` + workspace hygiene · Focus: single-source theme + tree hygiene_

## Verdict

The single-source-of-truth theme was **not** achieved. The token system (`tokens.ts`) exists and is well-designed, but the app barely uses it — most colors, spacing, fonts, and radii are hardcoded inline, and the brand palette is now defined in **three** competing places. Tree hygiene is mostly fine at the workspace level, but the `ember` package carries heavy dependency bloat and a couple of dangling references inherited from the Replit scaffold.

Nothing here is broken at runtime; these are maintainability/consistency problems. They're cheap to fix now and expensive later.

---

## 1. Theme is not single-source — it's triple-source

The same brand colors are declared independently in three files, with no link between them:

| Source | What it defines | Example |
|---|---|---|
| `src/lib/config/tokens.ts` | TS objects: `colors`, `typography`, `spacing`, `shadows`, `borderRadius`, palettes | `colors.brand.ember = "#E8613A"` |
| `src/index.css` `:root` | CSS custom properties | `--ember: #E8613A` |
| `src/index.css` `@theme inline` + `body` | Tailwind theme vars + hardcoded body styles | `--color-background: #080808`, `background:#080808` |

`#E8613A` (ember), `#080808` (bg), and `#F0E8E0` (foreground) are each written out by hand in 2–3 of these, plus inline in components. Change the brand color and you must edit it in many unconnected spots.

### What's actually consumed vs. dead

Only **two** exports from `tokens.ts` are genuinely used by the app:

- `environmentPalette` — 3 imports (Dashboard, EnvironmentDisplay)
- `companionPalette` — 1 import (CompanionDisplay)
- `animation.transition` — 1 use (EnvironmentDisplay)

Everything else is **dead code**:

- `colors` — **0** references outside `tokens.ts`
- `typography` — **0** references
- `spacing` — **0** references
- `shadows` — **0** references
- `borderRadius` (the token) — **0** references (the 21 "matches" are all the inline CSS prop `borderRadius:` in JSX, unrelated)

The `:root { --ember … }` custom properties in `index.css` are referenced **0 times** anywhere (`var(--ember)` etc. is never used).

So the carefully-built design system is ~70% unused, and the parts of the UI it was meant to govern ignore it.

## 2. Hardcoded styling throughout components/pages

Visual values are written as inline literals rather than pulled from tokens. Counts of hardcoded hex colors:

| File | Hardcoded hex |
|---|---|
| `pages/Dashboard.tsx` | 9 |
| `pages/CreateJourney.tsx` | 9 |
| `components/CheckInPanel.tsx` | 8 |
| `components/MilestoneList.tsx` | 7 |
| `components/StreakDisplay.tsx` | 4 |
| `components/MilestoneToast.tsx` | 4 |
| `components/ScarList.tsx` | 2 |
| `pages/not-found.tsx` | 1 |
| **Total** | **~44** |

Beyond hex, there are **45+** hardcoded `padding`/`margin`/`gap` literals and inline `fontSize`/`borderRadius`/`fontWeight`/`letterSpacing` values that duplicate what `typography`, `spacing`, and `borderRadius` already define. Example (`Dashboard.tsx:80-114`): `color:"#E8613A"`, `color:"#F0E8E0"`, `fontSize:"11px"`, `borderRadius:"8px"`, `padding:"8px"` — every one of these has a token equivalent that isn't used.

There are also new color values appearing **only** inline that exist in no token at all (e.g. `#FFF8F4`, `#D4B090`, `#C84A4A`, `#A0A0A0`, `#0A0804`, and alpha hexes like `#E8613A55`), so the palette is silently drifting wider than the documented system.

## 3. Tree hygiene

**Workspace structure is sound.** pnpm workspaces are organized cleanly: `artifacts/*` (apps), `lib/*` (shared packages), `scripts`. Package names are consistent (`@workspace/*`), tsconfig project references exist, and the engine/store/config separation described in `replit.md` is real and well done.

**Issues:**

- **Dependency bloat in `artifacts/ember/package.json`.** It declares ~50 dependencies — the entire shadcn/ui + Radix stack (~25 `@radix-ui/*` packages), plus `recharts`, `embla-carousel-react`, `cmdk`, `vaul`, `sonner`, `react-hook-form`, `@hookform/resolvers`, `react-day-picker`, `next-themes`, `framer-motion`, `lucide-react`, `date-fns`, `@tanstack/react-query`, `tw-animate-css`. The app's source imports **only** `react`, `react-dom`, `wouter`, `zustand`. **Zero** of the heavy libraries are used. This was copied wholesale from the mockup-sandbox scaffold.
- **Dangling alias.** `vite.config.ts:52` defines `@assets → ../../attached_assets`, but that directory doesn't exist. Harmless today (nothing imports `@assets`), but it's a trap.
- **`mockup-sandbox` is an unpruned scaffold.** 55 shadcn `ui/*` components + a 146-line demo `App.tsx` + `.generated/`. It's not referenced by `ember` or `api-server`. Fine to keep as a component sandbox, but it's dead weight if you don't intend to use it — and it's almost certainly the source the ember `package.json` deps were cloned from.
- **Backend packages are pure scaffold** (`lib/db`, `lib/api-zod`, `lib/api-client-react`, `api-server`). Consistent with "backend not yet wired" — flagging only so it's a conscious choice, not forgotten.

## 4. Build / typecheck health

Not run to completion — `node_modules` isn't installed in this audit environment and a full `pnpm install` (with the 1-day `minimumReleaseAge` gate) wasn't feasible here. Static checks passed: all local `@/`-aliased imports resolve to existing files, routing in `App.tsx` is coherent, and no source file imports the dangling `@assets` alias. Recommend running `pnpm install && pnpm run typecheck` locally to confirm.

---

## Recommendations (prioritized)

**P1 — Make the theme actually single-source.**
1. Pick `tokens.ts` as the one source. Delete the duplicated `:root { --ember … }` block in `index.css` (it's unused anyway), and either derive the `@theme`/`body` values from tokens or accept `index.css` holds only the two bootstrap colors and document that.
2. Replace inline hardcoded values in components/pages with token references (`colors.brand.ember`, `typography.fontSize.*`, `spacing.*`, `borderRadius.*`). Start with the two worst offenders, `Dashboard.tsx` and `CreateJourney.tsx`.
3. Fold the inline-only colors (`#FFF8F4`, `#C84A4A`, etc.) into `tokens.ts` so the palette is complete and intentional.

**P2 — Cut dependency bloat.** Strip `ember/package.json` down to what's imported (`react`, `react-dom`, `wouter`, `zustand`, plus build tooling). Removes ~45 unused packages and shrinks the install/attack surface. If you want shadcn components later, add them back per-component.

**P3 — Housekeeping.** Remove or fix the `@assets` alias. Decide whether `mockup-sandbox` stays (keep as a real sandbox) or goes (delete). Run `pnpm run typecheck` to confirm green.

**Optional — Enforce it.** Once migrated, an ESLint rule banning hex literals / raw px in `.tsx` style objects (or a lint script grepping for `#[0-9A-Fa-f]{3,8}` in components) keeps drift from creeping back.
